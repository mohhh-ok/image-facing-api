# 目的・スコープ・用語

## 目的

画像内の主たる被写体（キャラ・人・動物・物）が、**見る人から見て left / right のどちらを向いているか**
を判定する HTTP サービス。判定は学習可能で、正解ラベルを与えるほど精度が上がる。

最初のユースケースは [ai-kyoto-osaka](../../ai-kyoto-osaka) の妖怪姿絵を「向き合わせ」して合戦画面で
対面させること。だが本サービスは **特定ドメインに依存しない汎用判定器**として設計する
（project キーでドメインを分離する → [multi-tenant.md](multi-tenant.md)）。

## left / right の定義（重要・全体で統一）

**「viewer（画像を見ている人）から見た向き」** で統一する。

- `left`  = 被写体が **画面の左方向**を向いている（顔・体が左へ）。
- `right` = 被写体が **画面の右方向**を向いている。

判定の優先順位（被写体がキャラ/生き物の場合の推奨ルール）:

1. **顔（鼻先・視線）**。獣・鳥・妖怪では嘴・吻・口先が「鼻先」に相当する。
   両目が見えていても口先が片側へ突き出ていれば、その側を向いた 3/4。
2. 顔が正面なら **胴体の半身**（胸・襟・帯・前掛けの向き）。
3. **向きの根拠にしないもの**: 体の傾き/ねじれ、髪・尻尾・衣の流れ、踏み出した足、伸ばした腕、
   左右非対称な装飾。これらはポーズ・意匠であって向きではない。
4. 完全な正面顔は本来 left/right が存在しない。二択を強制する運用なら、主役の小道具を持つ手の
   画面位置など弱い根拠で機械的に倒し、**確信度を低く**返す。

> この優先順位は ai-kyoto-osaka の `docs/direction-judgement.md`（妖怪姿絵向けに磨いた判定基準）に由来する。
> 本サービスは基準をハードコードせず **ラベル（人が与えた正解）で学習する**ので、最終的な「何を left とするか」は
> その project のラベル分布が決める。上記は初期ラベルを付けるときの指針として使う。

## スコープ

- 入力 1 枚に対する **left / right の二値判定** と confidence。
- 正解ラベルの登録と、それによる即時の精度改善（k-NN）。
- project 単位のテナント分離。
- admin による手修正 UI。

## 非スコープ（やらないこと）

- `front` / 多クラス / 角度（yaw 度数）の回帰。**二値のみ**（confidence で曖昧さは表現する）。
  ※将来 front を足す余地は残すが、初版は二値。
- 顔検出・骨格検出・物体検出そのもの（埋め込みは画像全体に対してかける）。
- 画像の生成・加工・反転保存（**向きはデータとして返すだけ**。画像を反転して返さない。
  反転表示はクライアント側の責務 → [client-integration.md](client-integration.md)）。
- LLM 呼び出し（このサービスは LLM を持たない）。

## 設計判断の背景（なぜこの形か）

- **なぜ CNN を fine-tune しないか**: 少データ・CPU・即時反映を満たせないため。詳細 [model.md](model.md)。
- **なぜ独立サービスか**: 左右判定を複数サービスで再利用したいため（汎用化が目的）。
- **なぜ画像を反転して返さないか**: バイナリ反転は不可逆でやり直しが効かない。向きを「データ」で持てば、
  ラベルを直すだけで全クライアントの表示を後から修正でき、再生成も要らない。
