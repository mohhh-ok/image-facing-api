# Persistence and DB schema

A single SQLite file (`/data/facing.db`, WAL mode). All SQL is consolidated in `app/db.py`. Label images are downscaled to a long-side 256px JPEG and stored at `/data/images/<sha256>.jpg` (admin display only — re-embedding is served by the stored vectors in the `embeddings` table). The DB holds only paths/hashes, not the image bytes themselves.

## Why SQLite

- Best fit for a single-process, single-volume layout. Operationally lightweight.
- At the expected scale (thousands to tens of thousands of labels per project), performance is more than enough. Nearest-neighbor search over embeddings runs in memory, so the DB's main job is "persistence and listing."
- If tenants explode in number in the future, there is room to move to Postgres (the `db.py` consolidation limits the surface area for that swap).

## Tables

### projects

| Column | Type | Description |
|---|---|---|
| id | INTEGER PK | |
| name | TEXT UNIQUE | project key (appears in URLs; alphanumeric + hyphen) |
| description | TEXT | optional |
| api_key_hash | TEXT | hash of the API key (plaintext is not stored; see [security.md](security.md)) |
| k | INTEGER | k for k-NN (default 9; NULL means use the global default) |
| settings_json | TEXT | per-project settings such as transparent composite color, thresholds (optional) |
| created_at | TEXT | ISO8601 |

### samples (one labeled data point = one entry in the training set)

| Column | Type | Description |
|---|---|---|
| id | INTEGER PK | |
| project | TEXT | projects.name (indexed) |
| image_sha256 | TEXT | hash of the source image (rows with `is_flip_aug=1` point to the same sha as the original) |
| facing | TEXT | `'left' \| 'right'` |
| source | TEXT | `'human' \| 'import' \| 'model'`, etc. `human` has top priority |
| is_flip_aug | INTEGER | 0/1. 1 = a reverse-label row auto-generated by flip augmentation |
| origin_sample_id | INTEGER | the sample.id the flip row was derived from (for deduplication and regeneration). NULL on original rows |
| external_id | TEXT | client-side identifier (optional; stored only — matching is by image_sha256) |
| created_at | TEXT | |
| updated_at | TEXT | timestamp of the last facing update |

Indexes: `(project, image_sha256)`, `(project, facing)`, `(project, is_flip_aug)`.

### embeddings (embedding vectors)

| Column | Type | Description |
|---|---|---|
| sample_id | INTEGER PK FK→samples.id | |
| model_name | TEXT | e.g. `dinov2_vits14` |
| embed_version | INTEGER | version including preprocessing. Changing this triggers re-embedding |
| dim | INTEGER | 384 |
| vector | BLOB | L2-normalized float32 as raw bytes (np.tobytes) |

> samples and embeddings are split so that when the embedding model changes, only embeddings need to be rebuilt.
> One vector per sample (for the current model). If multiple models need to coexist, make `(sample_id, model_name)` the PK.

### predictions (optional audit log)

If predict requests are recorded, keep it minimal (do not store the image, just the hash). Useful for post-hoc accuracy analysis and for monitoring "how many uncertain are we returning." The first version may skip this.

| image_sha256 | TEXT | (when stored) |
| facing | TEXT | the returned result |
| confidence | REAL | |
| uncertain | INTEGER | |

## In-memory index

- At startup, load `embeddings` into a per-project numpy matrix (`store.py`).
- predict uses a matrix dot product (cosine) to find neighbors. label writes the DB row and appends one row to the matrix.
- The DB can rebuild the matrix on process restart, so memory is purely a cache.

## Migrations

- For the first version, `db.py`'s `init_schema()` just runs `CREATE TABLE IF NOT EXISTS`.
- Schema changes can use plain ALTER, or — since row counts are small — recreate and re-ingest via `scripts/import_labels.py`.
- Do not introduce a heavy ORM; the standard `sqlite3` module is enough.
